Swift Keyword Categories

Swift keywords fall into three categories based on when they can act as identifiers. Each category shown with two examples: one as keyword, one as identifier (where allowed).

Hard reserved — always keywords, never identifiers
Reserved — keywords, but identifiers in parameter labels & dot access
Context-specific — identifiers everywhere except their grammar context
Dotted underline — keyword acting as identifier

Category 1 — Hard Reserved

var · let · inout

Always keywords. Cannot be used as identifiers — not even as parameter labels — unless escaped with backticks.

Example 1 — as keywords (normal use)
let name = "Swift" var count = 0 func increment(_ value: inout Int) { value += 1 }
Example 2 — as identifiers (requires backticks!)
// Without backticks → compile error // func configure(var: String) { } ✗ error // func configure(let: String) { } ✗ error // Backticks required: func configure(`var`: String, `let`: Int) { print(`var`, `let`) }
Only 3 keywords in this category. They modify parameter bindings, so allowing them as parameter labels would create genuine syntactic ambiguity.

Category 2 — Reserved Keywords

func · class · return · if · …

Reserved as keywords in general, but can be used as identifiers in two cases without backticks: as parameter labels in function declarations/calls, and when accessed via dot notation as member names.

Example 1 — as keywords (normal use)
class Parser { func parse() throws -> Node { guard let token = nextToken() else { throw ParseError.unexpected } switch token { case .eof: return Node.empty default: return try parseExpression(token) } } }
Example 2 — as identifiers (parameter labels & dot access)
// Reserved keywords as parameter labels — no backticks needed! func move(from: Point, to: Point) { } func fetch(if: Bool, return: String, default: Int) { print(`return`) // inside body, backtick needed to refer to parameter } // As member name via dot notation — no backticks needed let center = NotificationCenter.default
This is the largest category. The parameter label exception exists because SE-0001 found ~200 imported APIs conflicted with keywords used as argument labels.

Category 3 — Context-Specific Keywords

mutating · lazy · get · set · …

Only keywords in the specific grammar context where they appear. Everywhere else, they are ordinary identifiers — no backticks needed.

Example 1 — as keywords (in their grammar context)
struct Counter { lazy var formatter = NumberFormatter() ``` var value: Int = 0 { willSet { print("will change") } didSet { print("did change") } } mutating func increment() { value += 1 } ``` } protocol Configurable { associativity // keyword in precedencegroup optional func reset() // keyword in @objc protocol }
Example 2 — as identifiers (outside their grammar context)
// Perfectly valid — these are just identifiers here let mutating = true let lazy = “not really” let weak = 42 var override = false var convenience = “store” // Using context-specific keywords as function and parameter names func set(dynamic: Bool, final: Int) { print(dynamic, final) } // Even as a type name struct lazy { var none: String = “something” }
This list keeps growing: newer additions include actor, async, await, borrowing, consuming, sending, nonisolated, isolated, some, any, macro, and package.
